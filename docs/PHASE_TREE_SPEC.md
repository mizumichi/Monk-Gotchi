# PHASE_TREE — キャラ進化から「木の育成 + 果樹園」への方針転換

> このドキュメントは `docs/PHASE_TREE_SPEC.md` として配置する。
> Claude Code はこの仕様を正としてフェーズ単位で実装すること。
> 各フェーズは「確認ポイント」を満たした時点で完了とし、次へ進む。

---

## 0. 方針転換の要約

23種のキャラ進化システムは管理負荷が大きいため廃止し、**1本の指標（7日間の累計スコア）で育つ木**に置き換える。

- 木は累計スコアに応じて**なめらかに連続成長**する（毎日の手応え）。
- 7日目に累計スコアで**3段階のランク**が決まり、ランクに応じて**実の数**が変わる（収穫）。
- 収穫した実は**果樹園**に累積し、継続のモチベーションになる。

**活かす資産**: `evolution.ts` の累計スコア計算・SlothDay 検出ロジック、`CharacterDex`（→ `Harvest` に転用）、E-4 の進化演出（→「実がなる演出」に転用)。
**廃止する資産**: `characters.ts` の23種、中期/最終キャラ判定、E-5（残り19体スプライト）。

## 1. 評価ロジック（getTreeRank）

`src/lib/evolution.ts` の中期・最終判定を撤去し、以下の純関数に置き換える。

```ts
export type TreeRank = 'low' | 'mid' | 'high';

export interface TreeResult {
  rank: TreeRank;
  fruitCount: number;
}

// score = 7日間の累計XP（既存の累計スコア計算をそのまま流用）
export function getTreeRank(score: number): TreeResult {
  if (score < 350) return { rank: 'low',  fruitCount: 1 };          // いまいち
  if (score < 700) return { rank: 'mid',  fruitCount: 3 };          // 普通
  // だいぶ良い: 700で6個 → 1050で12個（連続増）
  const f = Math.min(12, Math.round(6 + ((score - 700) / 350) * 6));
  return { rank: 'high', fruitCount: f };
}
```

**閾値（350 / 700）と実の数は暫定値。** 実プレイ後にこの関数の数値だけ調整する設計にする（調整箇所をここ1つに集約）。

ラベル対応: low=「いまいち」/ mid=「普通」/ high=「だいぶ良い」。

## 2. 木の成長ビジュアル仕様（TreeDisplay）

`CharacterDisplay` を `TreeDisplay`（SVG）に置き換える。プロシージャル生成・絵本風。

- props: `score: number`（現在の累計スコア）, 任意で `fruitCount`（収穫確定後に実数を固定表示したい場合）。
- 育成中は `score` から実数も連続表示してよい（収穫前のプレビュー）。確定収穫時は `fruitCount` を渡して固定。

成長パラメータ（`t = clamp(score / 1050, 0, 1)`、viewBox 320×360、CX=160、GROUND=300）:

| 要素 | 計算式 |
|---|---|
| 幹の高さ trunkH | `lerp(38, 150, t)` |
| 樹冠半径 crownR | `lerp(26, 96, t)` |
| 幹の根元半幅 bw | `lerp(7, 15, t)`（上端半幅 tw = bw*0.55) |
| 葉の塊の表示数 nBlobs | `round(lerp(3, 9, t))` |
| 実の半径 | `max(3, crownR * 0.085)` |

色（ダークモードでも視認できる中間トーン。固定値でよい）:
- 幹 `#7A5230` / 葉・奥 `#3B6D11` / 葉・手前 `#5A9E2E` / 実 `#E24B4A` / 地面 `#639922`

描画手順:
1. 地面の楕円（`cy=GROUND+6`, `rx=lerp(60,118,t)`, `ry=14`）
2. 幹のパス（根元から上端へ、左右をベジェで軽くくびれさせる台形）
3. 葉の塊（奥トーンを +4px 下にずらして描き、その上に手前トーンを重ねて陰影を出す）
4. 実（樹冠中心を基準に配置）

塊と実の相対配置オフセットは下記を使う（樹冠半径 crownR でスケール）:

```ts
const blobs = [[0,0,1],[-0.55,0.05,0.72],[0.55,0.05,0.72],[-0.32,-0.42,0.7],
  [0.32,-0.42,0.7],[0,-0.58,0.66],[-0.62,-0.28,0.55],[0.62,-0.28,0.55],[0.05,0.42,0.7]];
const fruitOff = [[0,-0.1],[-0.45,0.12],[0.45,0],[0.12,0.4],[-0.25,-0.38],[0.35,-0.32],
  [-0.58,-0.12],[0.55,0.28],[-0.1,0.18],[0.26,0.5],[-0.36,0.42],[0,-0.5]];
```

塊中心 = `(CX + blob[0]*crownR, crownCY + blob[1]*crownR)`、塊半径 = `blob[2]*crownR*0.54`（奥は *0.58）。
樹冠中心 `crownCY = (GROUND - trunkH) - crownR*0.18`。
実は `fruitOff` の先頭 `fruitCount` 個を使用。

> 既存のプロトタイプSVGロジックがそのまま土台になる。後でより描き込んだイラストSVGに差し替えても、props（score / fruitCount）の契約は変えないこと。

## 3. データモデル変更（amplify/data/resource.ts）

### Tree（既存 Character を簡略化）
- `cycleStartDate: string`（必須）
- 累計スコアは当該サイクルの `DailyLog` から都度計算する（保持しない）。
- **撤去**: `stage` / `midType` / `finalType` 関連フィールドと、それに依存するコード。

### Harvest（既存 CharacterDex を転用・リネーム）
- `owner`
- `harvestedAt: datetime`
- `cycleStartDate: string`
- `totalScore: integer`
- `rank: string`（'low' | 'mid' | 'high'）
- `fruitCount: integer`

**果樹園の合計実数 = 全 Harvest の `fruitCount` の総和。**

> DynamoDB はスキーマレスなので既存データは温存されるが、`data/resource.ts` の定義と型は上記に整理すること。`list()` はデフォルト100件上限のため、Harvest 件数が増えた場合のページネーションは別途対応（既知課題のまま据え置き可)。

## 4. 画面構成

### ダッシュボード（既存 /dashboard を改修）
- 上部: `TreeDisplay`（育成中の木。現在の累計スコアで連続成長）
- 現在の累計スコア・推定ランク・推定実数の表示
- 既存のタスク一覧・スコアバー・ジャーナルはそのまま
- Day 7 到達後に「収穫する」アクション → Harvest 作成 → 木をリセット（次サイクルへ）

### 果樹園ページ（/dex を /orchard にリネーム新設）
- これまで収穫した実の合計数
- サイクル履歴（収穫日・ランク・実数の一覧）
- ヘッダーのリンクを「📖 図鑑」→「🌳 果樹園」に変更

## 5. フェーズ分割（各フェーズで1つの確認ポイント）

| フェーズ | 内容 | 確認ポイント |
|---|---|---|
| **T-1** | `TreeDisplay`（SVG）実装。`CharacterDisplay` を置換 | スコアを変えると木が連続的に育ち、実数だけ変わる |
| **T-2** | `evolution.ts` を `getTreeRank` に差し替え。中期/最終判定を撤去。ダッシュボードにランク・実数表示 | Day 7 にランクと実数が正しく算出される |
| **T-3** | `Harvest` モデル + 収穫フロー + `/orchard` 画面。図鑑→果樹園 | 収穫すると果樹園の合計実数が増え、木がリセットされる |
| **T-4** | 収穫時の「実がなる演出」（E-4 の進化演出を転用） | Day 7 収穫時にアニメーションが再生される |
| **T-5** | 旧資産の撤去（`characters.ts` の23種、未使用コンポーネント、旧 `/dex`）。ビルド整理 | ビルドが通り、未使用コードが残っていない |

T-1 → T-5 の順。各フェーズのプロンプトは本仕様を参照しつつ、160行程度の簡潔な指示として別途用意する。

## 6. 留意点

- 既存の「+1日 / リセット」テストボタンと日付オーバーライド機構はそのまま活用し、木の成長と収穫を動作確認する。
- 進化演出の細部（フラッシュ→実が膨らむ→きらめき等）は、実装前にプロトタイプで尺・動きを確認してから T-4 のプロンプトを書く。
- 閾値・実数・成長パラメータは「見た目で気持ちよく感じる値」へ実プレイ後に調整する前提。変更は本仕様の数値を更新して反映する。
