# FRIENDS_SPEC.md — フレンド機能 設計仕様

> Phase D-1。このアプリ参加者全員のステータスを一覧表示するフレンド機能の確定仕様。
> 実装プロンプトは `PROMPT_FRIENDS.md`。このファイルは設計判断の根拠を残すためのもの。

---

## 1. 目的とスコープ

このアプリは友人数人だけが使う前提。そのため「友人検索」「申請・承認」といった相互フローは作らない。
**参加者全員が自動的に互いのステータスを見られる**（各自が公開設定をオンにしている場合）。

### やること
- 参加者全員のステータス一覧画面（`/friends`）
- ニックネーム（表示名）の設定・変更
- ステータス公開 / 非公開の切り替え

### やらないこと（このフェーズでは）
- 友人申請・承認・ブロック
- 個別チャットやリアクション
- 個人の生ログ（DailyLog など）の共有

---

## 2. 表示内容

フレンド画面では各ユーザーについて以下を表示する：

| 項目 | 内容 |
|---|---|
| ミニ木アイコン | stage と todayXp で育ち具合がビジュアル変化 |
| ニックネーム | 自分のカードには「あなた」バッジ |
| ステージ + 日数 | 例: `🌳 成長中　Day 4 / 7` |
| 今日のXP | 数値 + バー |
| 累計果実数 | 🍎 + 数値 |

### 公開する値の出どころ
ユーザーが選んだのは「今日のXP / 木の育ち具合(ステージ・サイクル日数) / 累計果実数」。
これらは集計済みの値であり、個人の生ログは含めない。

---

## 3. XPバーの表示（厳密な上限なし）

XPに固定の上限は存在しない（タスクの組み合わせで青天井になりうる）。
したがって `/ 150` のような分母表記はしない。バーは見た目のためだけにソフトキャップ式で伸ばす：

```ts
const xpBarPct = (xp: number) => {
  if (!xp || xp <= 0) return 0;
  return Math.min(100 * (1 - Math.exp(-xp / 180)), 98);
};
```

参考値: 50XP→24%、100XP→43%、150XP→57%、300XP→81%。
数値が増えるほど伸びが緩やかになり、100%には到達しない。表示は「数値 + バー」のみ。

---

## 4. ソート順（確定）

1. 公開ユーザーを `todayXp` の降順で並べる
2. **非公開ユーザーは XP に関わらず常に末尾に固定**

非公開ユーザーのカードは鍵アイコン + 「非公開ユーザー」のみ。
**名前もスコアも表示しない。**

## 5. ニックネーム

### 既定値
未設定のまま保存した場合は **"ぬる"** を既定ニックネームとする。

### 文字数
最大 16 文字。入力中は文字数カウンタを表示。

### 全員公開の明示（確定文言）
設定モーダルに以下の注意書きを必ず出す：

> ニックネームはこのアプリに参加している**全員に公開**されます。本名や個人情報は入れないでください。

### バリデーション
`src/lib/nicknameValidation.ts` に切り出し、モーダルと設定パネルの両方で使う。
性的・差別・暴力系のワード、および記号・空白のみの文字列を弾く。
NG時は入力枠を赤くし、エラー文を表示し、保存ボタンを無効化する。

これはクライアント側の一次フィルタであり、完璧な検閲ではない。
「うっかり / 悪ふざけ」を防ぐのが目的。NGリストの拡充は後日でよい。

## 6. 設定フロー

### 初回モーダル
フレンド画面に**初めて入ったタイミングでのみ**ニックネーム設定モーダルを出す。
`User.hasSeenFriendsIntro` が false のときに表示し、保存時に true にする。
（ダッシュボードや図鑑など他画面では出さない。フレンド画面に限定。）

モーダルの構成：ニックネーム入力 → バリデーション → 全員公開の注意書き → 公開トグル（既定オン）→ 保存。

### 設定パネル（再編集）
フレンド画面ヘッダー右の「⚙️ 設定」ボタンで、**画面内にインライン展開**するパネルを開く（別ページに遷移しない）。
ニックネーム変更・公開設定の切り替えができる。同じバリデーションを適用。

## 7. データモデル

### User（既存モデルに追加）
本人だけが読み書きする「正」のデータ。

```
nickname:            a.string()    // 表示名。未設定時は "ぬる"
isStatusPublic:      a.boolean()   // 公開フラグ。既定 true
hasSeenFriendsIntro: a.boolean()   // 初回モーダル表示済みか。既定 false
```

### PublicProfile（新設・一覧表示専用）
参加者全員が read できる公開モデル。**owner本人だけが書き込み可。**

```
nickname:       a.string()
isStatusPublic: a.boolean()
todayXp:        a.integer()
cycleDay:       a.integer()
stage:          a.string()    // egg / early / mid / final
totalFruits:    a.integer()
// owner / updatedAt は Amplify が自動付与
```

認可：
```
PublicProfile: allow.authenticated().to(['read']),
               allow.owner().to(['create', 'update', 'delete'])
```

### 公開方式の決定理由（案A採用）
全員のステータスを一覧するには、自分のレコードしか読めない既存の userPool 認可では不足する。
検討した2案：

- **案A（採用）**: 公開専用モデル `PublicProfile` を新設。集計済みの最小フィールドのみを全員readにする。
- 案B（不採用）: User モデル自体を全員read可にする。実装は楽だがUser全体が露出するため却下。

案Aを採用した理由は**最小公開の原則**。個人の生ログ（DailyLog など）を一切露出させず、表示に必要な集計値だけを別テーブルに切り出すことで、プライバシー面の安全性が高い。

### ミラー書き込み
ダッシュボードでスコアが更新されるタイミングで、本人の `PublicProfile` に
`todayXp` / `cycleDay` / `stage` / `totalFruits` をミラー書き込みする。
`nickname` / `isStatusPublic` は設定変更時に同期する。

### 非公開ユーザーの扱い
`isStatusPublic = false` のレコードも read 自体は可能。
まずは**クライアント側で中身を伏せて「非公開カード」に置き換える**方式とする。
（サーバー側で非公開ユーザーの中身を返さない設計の方が理想だが、本フェーズではクライアントマスクで可。）

## 9. プライバシー原則（恒久ルール）

- **個人の生ログ（DailyLog 等）は他ユーザーに公開しない。** フレンド機能で共有するのは `PublicProfile` の集計済みフィールドのみ。
- **ニックネームは全ユーザーに公開される表示名。** 新しい入力経路を追加するときも、本名・個人情報を弾くバリデーションを前提とする。

この2点は `CLAUDE.md` にも反映する。
